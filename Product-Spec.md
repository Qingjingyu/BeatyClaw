# Agentic Product Spec

## Goal

Agentic 第一期要做成苏白自己的单用户 AI 工作台。

它不是第一天就对外售卖的 SaaS，也不是让用户自己部署 Agent 的开源控制台。第一期目标是把现有 `Yoyoo0.1 / hermes-web-ui` 的页面结构和设计收口成 Agentic 工作台，并接入我们已经部署好的 Coco / Zylos / hxa-connect 后端能力。

核心原则：

- 页面结构沿用 Yoyoo0.1 现有设计，不重新设计整套 UI。
- 模型 API key、Zylos runtime、hxa-connect token、后端运行环境由平台内置。
- 用户只需要在“链接 / 频道”里填写对应平台的连接信息，例如飞书、Telegram、微信、企业微信等。
- 第一期只服务单用户，也就是苏白自己的工作台。

## Target Users

第一期目标用户只有一个：苏白。

苏白需要一个统一工作台来完成：

- 和 Agent 对话。
- 查看历史会话。
- 把想法沉淀成任务。
- 用看板跟踪任务状态。
- 配置外部渠道连接。
- 查看技能、记忆和用量。

## Problem

当前我们已经有几块能力，但它们是分散的：

- Zylos Web Console 可以聊天，但页面和产品体验弱。
- hxa-connect 已经证明多 Agent 通信链路可用，但没有工作台入口。
- Zylos 官方组件支持飞书、微信、Telegram、Slack、WhatsApp、钉钉等渠道，但需要配置和启用。
- Yoyoo0.1 / hermes-web-ui 有完整工作台页面，但还需要改名、收口和接入我们的 Coco / Zylos 后端。

Agentic 第一期要解决的是：把这些能力整理成一个能真正使用的单用户工作台，而不是继续停留在部署日志、命令行和零散页面里。

## MVP Scope

### 1. 登录

保留单用户登录。

第一期不做注册、不做邀请、不做多用户管理。登录只用于保护工作台入口。

验收标准：

- 未登录不能进入工作台。
- 登录后默认进入对话页。
- 登出后回到登录页。

### 2. 对话

沿用现有对话页设计。

对话后端逐步从 Hermes API 适配到 Agentic API，再接入 Zylos / Coco runtime。第一期允许先保留 Hermes 兼容接口，但产品命名和边界必须收口到 Agentic。

验收标准：

- 能发送消息。
- 能看到助手回复。
- 支持流式或近实时反馈。
- 消息失败时有错误提示。

### 3. 历史

沿用现有历史 / 会话列表能力。

验收标准：

- 能查看历史会话。
- 能搜索历史会话。
- 能从历史进入对应会话。

### 4. 任务

第一期任务模块用于记录和管理待办任务，不追求完整项目管理系统。

验收标准：

- 能创建任务。
- 能查看任务列表。
- 能编辑、暂停、恢复或删除任务，具体取决于现有能力。
- 任务与对话、看板的打通可以分阶段完成。

### 5. 看板

沿用现有 Kanban 页面和数据结构。

验收标准：

- 能查看任务卡片。
- 能按状态分列。
- 能创建、更新、完成、阻塞任务。
- 能查看任务详情和评论。

### 6. 链接 / 频道

这是 Agentic 的关键产品入口。

用户不需要填写模型 API key，只需要填写外部渠道连接信息。

第一期可展示现有代码支持的渠道，按实际可接通程度分级：

- 可配置：已有配置字段和组件代码。
- 可验证：能通过后端检查凭证或连接状态。
- 已连接：组件启动并能收发消息。
- 实验性：有代码但稳定性或平台风险较高。

优先渠道：

1. 飞书 / Feishu
2. Telegram
3. 企业微信 / WeCom
4. 微信个人号 / WeChat，标为实验性
5. Slack / WhatsApp / 钉钉，按现有组件成熟度逐步接入

验收标准：

- 页面展示可连接渠道。
- 每个渠道说明需要填写什么。
- 用户填写后由后端保存配置。
- 后端不把密钥暴露给前端。
- 至少一个渠道能完成配置保存和状态展示。

### 7. 技能

沿用现有技能浏览页面，后续适配 Zylos skills / components。

第一期不强求完整 skill 市场，但要能看见当前已安装技能或组件。

验收标准：

- 能查看技能列表。
- 能查看技能详情。
- 能区分已启用、未启用或未安装状态。

### 8. 记忆

沿用现有记忆页面。

验收标准：

- 能查看记忆内容。
- 能编辑并保存记忆。
- 记忆内容必须在服务端存储，不写入前端。

### 9. 用量

第一期用量先做工作台级统计，不做套餐账单。

验收标准：

- 能查看消息数、会话数、token 用量或估算 token。
- 能按模型或日期展示趋势。
- 如果真实 token 暂时无法完整采集，需要在页面和文档里标明是估算。

## V2 / Later

后续再做：

- 多用户 / 多租户。
- 注册和邀请。
- Stripe 或国内支付。
- 套餐和额度。
- 团队空间。
- 企业管理后台。
- 完整 skill 市场。
- 用户自带模型 key。
- 对外开放 SaaS。
- PostgreSQL 生产迁移。
- 完整审计日志和风控。

## User Stories

- 作为苏白，我想登录 Agentic 后直接看到对话页，这样我可以马上和 Agent 工作。
- 作为苏白，我想看到历史会话，这样我可以继续之前的上下文。
- 作为苏白，我想把事情放进任务和看板，这样我能追踪 AI 工作进度。
- 作为苏白，我想在链接页配置飞书或 Telegram，这样我能从外部聊天工具使用 Agent。
- 作为苏白，我想看到技能和记忆，这样我知道 Agent 会什么、记住了什么。
- 作为苏白，我想看到用量，这样我知道模型和消息消耗。

## Core Flows

### Flow 1: 登录进入工作台

1. 打开 Agentic。
2. 输入账号密码。
3. 登录成功。
4. 进入对话页。

### Flow 2: 对话和历史

1. 在对话页发送消息。
2. Agent 返回回复。
3. 会话保存进历史。
4. 从历史页搜索并打开旧会话。

### Flow 3: 任务和看板

1. 创建任务。
2. 任务进入看板。
3. 修改任务状态。
4. 查看任务详情、评论和执行记录。

### Flow 4: 配置链接 / 频道

1. 进入链接页。
2. 选择一个渠道。
3. 填写该渠道所需凭证。
4. 后端保存配置。
5. 页面显示配置状态。
6. 后续组件启动后显示连接状态。

### Flow 5: 查看技能、记忆、用量

1. 进入技能页查看可用能力。
2. 进入记忆页查看和编辑 Agent 记忆。
3. 进入用量页查看消耗。

## Functional Requirements

- 必须保留 Yoyoo0.1 现有页面结构作为第一期 UI 基础。
- 必须把产品名收口为 Agentic。
- 必须支持单用户登录。
- 必须隐藏模型 API key，前端不得要求用户填写模型 key。
- 必须提供对话、历史、任务、看板、链接、技能、记忆、用量入口。
- 链接页必须面向用户可理解的渠道连接流程，而不是暴露底层配置文件。
- 后端必须负责保存渠道凭证和运行配置。
- 接入 Coco / Zylos / hxa-connect 的地方必须通过服务端适配层完成。

## Non-Functional Requirements

- 密钥不得写入前端代码、localStorage 或公开接口响应。
- 第一阶段可以继续使用 SQLite / 本地文件，但必须保留后续迁移 PostgreSQL 的边界。
- 所有高风险操作，例如启动组件、重启服务、写入密钥，必须有后端校验和日志。
- 页面必须保留基本 loading、empty、error 状态。
- 不做大规模 UI 重设计，避免消耗第一期时间。

## Open Questions

- 第一阶段对话后端先接 Hermes 兼容接口，还是直接接 Zylos Web Console / Codex runtime？
- 第一阶段最先打通哪个渠道：飞书还是 Telegram？
- 用量统计第一阶段是使用真实 token，还是先用估算 token？
- hxa-connect 在第一期只做后端能力，还是也在页面展示 Agent 在线状态？

## Acceptance Criteria

- `Product-Spec.md` 和 `DEV-PLAN.md` 已存在，并且范围一致。
- 工作台第一期明确为单用户，不包含多租户、支付和套餐。
- 页面清单包含登录、对话、历史、任务、看板、链接、技能、记忆、用量。
- 文档明确模型 key 和后端 runtime 内置，用户只配置渠道连接信息。
- 文档明确沿用 Yoyoo0.1 设计，不重做 UI。
- 文档明确接入 Coco / Zylos / hxa-connect 的方向和边界。
